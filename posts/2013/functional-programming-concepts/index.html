Functional Programming Concepts: What We Were Never Taught in School

Nov 22 2013Tags: scala functional programming

I remember one of my CS profs in college telling the class that when we graduate, we will come out knowing this much (petite two-finger gesture here) about programming. Well, I don’t know about that, as programming in the real world is considerably further up the curve on mechanics, tools reliance, and development fads, but far behind the curve on applying theory and good engineering practice. But there you have it. The academic view on Bachelor’s degree candidates. However, there some truth to this statement.

In my education, we were never taught outright functional programming methodologies - at least not enough from what I recall. I certainly never had a 100-level course called a survey of formal programming methodologies. Should I have? Almost all of my education was spent studying CS techniques, applications, languages and processes from the perspective of procedural and object-oriented design, under Waterfall process and CMM buiness methodology. Nothing of Agile development methods, no web/cloud programming coursework, no JavaScript/HTML. Everything was taught from an imperative programming approach with languages like C, C++ and Java, and performed in console applications. I feel deprived in a way, because although we were taught most of this stuff, it was never framed from the perspective of Pure Functional Programming.

Since I am in the process of learning Scala on my own now, I’ve experienced some “methodology shock” with the language - a discomfort in the ways of programming with Functional methods. Douglas Crockford, in his Google Tech Talk about Monads Monads and Gonads starts by defining Functional Programming as having two kinds of meanings.

Programming with Functions

The first meaning of Functional Programming, according to Crockford, is the simple abiity to program using functions. This entails the familiar C-like structures of storing sequential code in an enclosing block, whose entry point for data is defined by a parameter list, and whose result can be returned outside the block to the caller.

Some developments occurred to enhance this model: First Class Functions, High Order Functions and Lexical Closure.

First Class Functions are functions that can be treated like data. First-Class functions, like the Integer data type, can be assigned to fields, given as arguments to other functions, and stored in data structures.

High Order Functions, are functions that can operate on other functions as arguments.

Lexical Closure - An enclosed function has access to a variable in an outer function. A big breakthrough.

Pure Functional Programming

The second meaning of Functional Programming is Pure Functional Programming is where functions are treated the same as mathematical functions. Mutation of existing data is not allowed. There can be no side-effects (modifcations to internal or external data).

In Pure Functional design, immutable data can only be input to a function, calculation done on it, with some other different immutable data returned. Re-running the function with the same input will always result in the same output. No I/O can be performed.

The advantages are that pure functions can be treated as entities whose input is conceptually identical to static data. A function is-a value. Therefore, the distinction between functions and data are erased. They become First Class, one in the same thing. This leads to higher reliability, consistent, more testable correctness, and a kind of full simplicity.

The disadvantages are that if you make everything immutable, you won’t be able to do much of anything with a program (a sentiment echoed by Haskell guy Simon Peyton Jones In summary, immutability makes it hard to work with real-world programming problems where data changes.

But the saving chord of this is the loophole of High Order Functions functions, which allow immutable functions to have dynamic behavior. Combining two functions offers a kind of pseudo-mutability to Pure Functional programs. No side-effects, but things like IO can be performed. Thus gave birth to the concept of Monads, which is where I stop trying to make sense of this.
